# 外贸 Skill 总协作手册

## 项目定位

这个目录用于创作、维护和扩展适合外贸领域各业务节点的实用型 Skill。

这里的 Skill 不是为了展示技术，而是为了直接解决业务动作中的重复劳动、判断成本和协作断层。每个 Skill 都应尽量同时满足三件事：

- 有明确业务结果
- 能讲成课程案例
- 能被龙虾或 Codex 即插即用

当前母目录优先服务以下目标：

- 支持国内培训业务增长
- 持续验证外贸 AI Skill 的真实需求
- 沉淀一套可复制的外贸业务节点 Skill 方法论

当前母目录默认按三层发布结构推进：

1. 单节点 Skill
2. 链路组合包
3. 总 Skill / 总编排器

默认先把每个业务节点做成可独立安装、可独立演示、可独立交付的单点 Skill，再进入链路组合和总编排阶段。

## 服务对象

这套目录优先服务以下人群：

- 外贸企业老板
- 外贸业务员 / 客户开发人员
- 销售支持与运营人员
- 课程学员
- 需要把 AI 落地进真实业务流程的人

典型场景包括：

- 客户开发前的信息核验
- 线索筛查和优先级排序
- 开发信、报价和跟进文案辅助
- 竞品与市场动态监控
- 客户资料管理和待跟进提醒
- 围绕外贸业务的内容辅助和案例沉淀

## 协作对象与输出偏好

### 对龙虾 / OpenClaw

- 优先写清楚触发时机、输入格式、输出结构
- 明确哪些步骤适合自动化，哪些步骤必须人工复核
- 默认优先“能跑通的最小版本”，再考虑多代理编排
- 不假设拥有私有客户数据或未授权系统权限

### 对 Codex

- 优先写清目录结构、脚本入口、文档和示例输入
- 默认要求可调试、可改造、可替换搜索源或输出模板
- 复杂逻辑要拆成可读模块，不做难维护的大一坨脚本
- 关键行为要能通过示例或脚本结果验证

### 通用输出偏好

- 默认中文说明
- 对外销售内容、开发信或摘要可中英混合
- 输出优先 Markdown、JSON、CSV、可复用脚本
- 不产出空泛概念稿，尽量落到业务动作和可交付结果

## Skill 设计原则

每个 Skill 在立项和实现时，必须满足以下原则：

1. 必须解决一个明确业务节点问题
- 不能只做泛工具介绍
- 必须说明替代了什么重复劳动，或者提升了什么判断效率

2. 必须能定义最小输入和明确输出
- 输入允许稀疏，但不能模糊到无法执行
- 输出必须结构化，方便交接、复核或继续加工

3. 必须可验证
- 需要能从最小样例跑通
- 需要能通过示例输入验证行为，不靠“理论上可以”

4. 必须保守
- 不伪造事实
- 不捏造客户画像
- 不把推断写成确定事实
- 不暗示访问私有数据

5. 必须能课程化表达
- 能解释为什么做
- 能解释怎么用
- 能解释怎么改成自己的版本
- 能解释如何接到真实业务结果

6. 默认优先做即插即用版本
- 先做最小可用的单点 Skill
- 再考虑复杂自动化、批处理或多代理协作

7. 默认先做单节点发布，不直接跳总包
- 单节点 Skill 是默认发布单位
- 链路组合优先用于课程演示、业务闭环验证和安装复用
- 不直接跳做“大而全总 Skill”
- 后续总 Skill 的定位应是编排器，而不是替代各节点 Skill 的一体化实现

## Skill 标准结构

每个 Skill 子目录默认采用以下结构：

```text
skill-name/
├── SKILL.md
├── README.md
├── scripts/
├── examples/
└── references/
```

补充规则：

- 如果需要专门面向 OpenClaw / 龙虾，可并行维护 `for-openclaw/` 变体
- `SKILL.md` 负责调用说明和工作流
- `README.md` 负责人类阅读入口和目录说明
- `scripts/` 放脚本入口和实现逻辑
- `examples/` 放最小输入示例
- `references/` 放模板、评分规则、提示词依据等

每个 Skill 必须至少声明：

- 适用场景
- 输入格式
- 输出格式
- 核心规则
- 风险与边界
- 默认行为

## Skill 生命周期

Skill 的候选来源包括：

- 课程高频问题
- 客户真实需求
- 内容反馈
- 业务流程中的高频卡点

统一状态如下：

- `候选`
- `设计中`
- `开发中`
- `可演示`
- `可交付`
- `归档/暂停`

流程要求：

1. 新方向先进入 `skill需求池.md`
2. 用 `模板/skill立项模板.md` 形成最小方案
3. 确认输入输出、边界和验收方式后再开发
4. 开发完成后对照 `模板/skill验收清单.md` 验收

## 课程与内容联动规则

每个 Skill 都要能映射到三条链路：

1. 客户问题
2. Skill 方案
3. 课程表达 / 内容选题

如果一个方向无法同时支撑业务价值和课程表达，优先级应下降。

推荐判断方式：

- 这个 Skill 是否来自真实业务卡点
- 这个 Skill 是否能讲清“为什么这样设计”
- 这个 Skill 是否能成为公众号、视频号或课程案例
- 这个 Skill 是否能沉淀成长期可复用资产

补充协作要求：

- 每个节点 Skill 在立项时，除了“所属业务节点”，还应考虑它未来处于哪条业务链路中
- 设计时优先保证它能作为单节点发布单元存在，不默认依赖全链路才成立
- 如果要做链路组合，应优先复用现有单节点 Skill，而不是重写成另一套大流程
- 如果要做总 Skill，应把重点放在调度、输入输出衔接和人工复核点设计上

## 当前母目录优先级

首批重点方向固定为：

1. 客户背调 Skill
2. 开发信 Skill
3. 竞品监控 Skill
4. 客户管理 Skill
5. 跟进优先级 Skill

第二批储备方向包括：

- 展会线索筛选 Skill
- 询盘质量打分 Skill
- 报价单整理 Skill
- WhatsApp / 微信跟进辅助 Skill

## 协作边界

- 不为了堆数量而批量造 Skill
- 不在需求未立项前直接开写复杂实现
- 不把弱需求包装成高优先级项目
- 不跳过示例输入、边界说明和验收标准

## 新增 Skill 的默认步骤

1. 阅读本手册，确认是否符合母目录定位
2. 在 `skill需求池.md` 中登记候选需求
3. 复制并填写 `模板/skill立项模板.md`
4. 确认输入、输出、边界、依赖、课程映射，以及它未来属于哪条链路
5. 进入对应子目录开始实现
6. 用 `模板/skill验收清单.md` 完成验收
