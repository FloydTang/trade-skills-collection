# 外贸 Skill 合集

这个目录用于统一孵化、维护和沉淀适合外贸业务各节点的实用型 Skill。

它不是单一 Skill 仓库，而是一个面向业务结果、课程表达、发布复用和长期演进的 Skill 合集仓库。这个仓库默认承担“分发与演示仓库”角色，当前优先服务两类协作对象：

- 龙虾 / OpenClaw：偏云端调用、自动化、多代理协作
- Codex：偏本地开发、调试、文档沉淀和二次改造

## 当前重点说明

当前请优先把它理解为“4 个已完成节点 + 1 个基于这 4 个节点的组合包”的仓库。

当前优先维护、测试、验证和持续修补的对象只有：

- `客户搜索skill`
- `线索整理skill`
- `客户背调skill`
- `开发信skill`
- `主动开发链路组合包`

其他目录当前主要是需求占位、方向保留或后续规划入口，不代表已经完成、已经验收或推荐优先安装。

补充一条协作边界：

- 当前仓库是 `外贸skill` 的真实源；Obsidian `工具工作间/外贸skill` 是对应的展现层，不是同一路径自动等同

如果你希望给 OpenClaw、Codex 或其他 agent 一个最直接的安装范围说明，请先看：

## 当前可直接使用的 Skill / 组合包

说明：

- 当前真正推荐优先安装和测试的，只有下面这 5 项
- 其中核心主线仍是 4 个已完成节点
- `主动开发链路组合包` 是基于这 4 个节点的组合使用入口

### 1. 主动开发最小闭环链路组合包

- 作用：把客户搜索、线索整理、客户背调、开发信串成一个可直接演示和安装的最小主动开发闭环
- 当前状态：可演示
- 适合场景：课程学员理解全链路、agent 用户快速安装试跑、团队内部演示、从单节点继续扩展组合包
- 合集目录：`主动开发链路组合包/`
- 当前特点：已提供固定样例、阶段化中间产物、最小串联脚本和公开运行说明
- 推荐入口：先看 `主动开发链路组合包/README.md`，再运行 `主动开发链路组合包/scripts/run_minimal_demo.py`

### 2. 客户背调 Skill

- 作用：把公司名、联系人、邮箱或官网整理成结构化客户情报报告
- 当前状态：可演示
- 适合场景：客户真假初判、官网真假初判、开发前置研究
- 合集目录：`客户背调skill/`
- 独立仓库：[trade-customer-intel](https://github.com/FloydTang/trade-customer-intel)

### 3. 客户搜索 / 线索发现 Skill

- 作用：用公开网页与 LinkedIn 结果线索发现第一批候选客户
- 当前状态：可交付
- 适合场景：主动开发入口、公开搜客户、候选名单构建
- 合集目录：`客户搜索skill/`
- 独立仓库：[trade-lead-discovery](https://github.com/FloydTang/trade-lead-discovery)

### 4. 开发信 Skill

- 作用：把客户基础信息、客户画像摘要和跟进阶段转成英文开发信草稿
- 当前状态：可交付
- 适合场景：首轮开发信、跟进邮件、背调结果转首轮触达动作
- 合集目录：`开发信skill/`
- 独立仓库：[trade-outreach-email](https://github.com/FloydTang/trade-outreach-email)

### 5. 线索整理 / 初筛 Skill

- 作用：把零散候选线索整理成可继续进入客户背调的标准输入
- 当前状态：可交付
- 适合场景：搜索结果整理、字段统一、缺失提示、初步分类
- 合集目录：`线索整理skill/`
- 独立仓库：[trade-lead-screening](https://github.com/FloydTang/trade-lead-screening)

## 单点使用与主表融合

`Lead Workflow Master` 的定位不是“只能记录全链路组合包运行”，而是“所有线索在当前外贸工作流中的总索引表”。

这意味着：

- 可以从全链路进入主表
- 也可以从单点 Skill 直接进入主表
- 外部导入、人工录入、展会名单、老客户表都可以接入同一张主表

推荐 `source_stage` 取值：

- `lead_discovery`
- `lead_screening`
- `customer_intel`
- `outreach_email`
- `external_import`
- `manual_entry`

典型情况：

- 如果今天你只单独跑客户背调，那么先创建或更新主表记录，再挂客户背调文档，`current_stage=customer_intel`
- 如果今天你只单独跑开发信，那么先创建或更新主表记录，再挂开发信文档，`current_stage=outreach_email`
- 如果后续再补跑前序或后序节点，不是新建平行 lead，而是回到同一条主记录继续补字段和资产链接

推荐的主记录合并顺序：

1. `email` 完全一致
2. `company_name + company_website` 基本一致
3. `company_name + person_name` 高度一致
4. 如果仍不确定，再人工确认是否合并

## 发布结构

这个母目录默认采用“三层结构”推进，而不是一开始就做一个大而全的总 Skill。

### 第一层：单节点 Skill

- 每个 `*skill/` 子目录默认先做成一个独立发布单元
- 它既是业务中的最小可用节点，也是课堂上最适合即插即用的安装单元
- 当前主发布节奏以这一层为准

### 第二层：链路组合包

- 当多个单节点 Skill 的输入输出稳定后，再把它们按业务顺序组合成轻量链路
- 这一层主要服务课程演示、业务闭环验证和安装复用
- 它不是新的重型 Skill，而是对已有节点 Skill 的组合使用

### 第三层：总 Skill / 总编排器

- 总 Skill 放在后续阶段推进
- 它的职责是调度和串联各节点 Skill，而不是替代各节点 Skill 的独立存在
- 只有当前两层稳定后，才进入这一层

### 版本策略

- Codex 版与 OpenClaw / 龙虾版属于运行形态差异
- 它们不改变“三层结构”的发布策略
- 默认先让每个节点 Skill 可单独安装、单独演示、单独交付
- 整体打包和总编排器放到后期统一推进

## 当前原则

- 优先做即插即用的单节点业务 Skill，不先追求复杂系统
- 优先服务真实客户问题、课程高频问题和内容反馈
- 当前优先确保 4 个已完成节点的测试、验证和持续修补，不并行扩张过多新 Skill
- 优先先发布单节点，再组织链路组合，最后才做总编排器
- 输出必须可复用、可验证、可继续改造
- 默认中文协作，对外销售内容允许中英混合
