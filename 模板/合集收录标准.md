# Skill 合集收录标准

这份文档用于判断一个单节点 Skill 是否可以进入合集仓库。

合集仓库的定位不是主开发仓库，而是：

- 对外分发仓库
- 一键下载入口
- 开源能力展示仓库
- 主动开发链路的组合展示仓库

因此，进入合集仓库的 Skill 不要求“功能已经完美”，但必须满足“用户拿到后能理解、能运行、能复用”的最低标准。

## 一、双轨发布原则

默认采用双轨结构：

1. 单个 Skill 独立仓库
2. Skill 合集仓库

默认分工：

- 独立仓库：该 Skill 的主开发源，允许更快迭代
- 合集仓库：收录稳定版本，方便演示、交付和一键下载

默认同步规则：

- 不是所有开发中的 Skill 都要立刻同步进合集仓库
- 只有达到“可演示”或“可交付”的节点 Skill，才进入合集仓库
- 合集仓库优先保证完整可下载，不依赖用户额外初始化 submodule
- 同步进合集时，应尽量同步对应的状态说明、样例和边界文档

## 二、最低收录条件

准备进入合集仓库的节点 Skill，至少满足以下条件：

1. 有 `README.md`
2. 有 `SKILL.md`
3. 有最小示例输入
4. 有明确输入格式
5. 有明确输出格式
6. 有核心规则说明
7. 有风险与边界说明
8. 有默认行为说明
9. 能从最小样例跑通
10. 目录结构清楚，便于继续改造
11. 如依赖其他 Skill，要写清上下游衔接关系
12. 默认中文说明，必要处允许中英混合
13. 明确标注当前状态
14. 明确标注作者信息

## 三、不应收录进合集的情况

以下情况原则上不进入合集仓库：

- 还没有立项完成
- 输入输出结构还不稳定
- 只有零散脚本，没有清楚说明
- 不能从最小样例跑通
- 风险边界不清楚
- 依赖大量口头解释才能使用
- 明显仍处于快速试错阶段

## 四、建议收录流程

建议按以下顺序判断：

1. 是否已经在 `skill需求池.md` 中完成立项
2. 是否已经用 `模板/skill立项模板.md` 收敛过最小方案
3. 是否已经满足本文件的最低收录条件
4. 是否至少达到“可演示”
5. 是否需要同步到合集仓库用于公开分发或统一展示

## 五、收录后的维护原则

- 合集仓库中的版本是“阶段性稳定副本”
- 独立仓库仍可继续快速演进
- 合集仓库不要求跟随每次小改动实时同步
- 当独立仓库出现重要增强、结构调整或状态变化时，再同步进合集仓库
- 同步时应补齐对应 README、样例、状态说明和链路衔接说明

## 六、当前推荐理解

- 独立仓库：偏开发与单点发布
- 合集仓库：偏交付、展示和一键下载
- 进入合集仓库，不代表功能完全封板
- 没进入合集仓库，也不代表方向无效，只表示还未达到收录时机
