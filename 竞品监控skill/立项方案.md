# 竞品监控 Skill 立项方案

## 基本信息

- Skill 名称：竞品监控 Skill
- 所属业务节点：市场研究 / 竞品跟踪
- 负责人：Codex + Floyd
- 当前状态：开发中

## 业务问题

- 这个 Skill 要解决的具体问题是什么：
  竞品官网价格、产品页、促销文案和上新动态经常变化，但人工反复打开网页查看效率低，且容易漏掉真正值得行动的变化。
- 当前人工是怎么做的：
  人工定期打开竞品网站，查看首页、产品页和促销页，靠截图、口头同步或零散笔记记录变化。
- 为什么值得被做成 Skill：
  这类动作重复、可结构化、业务解释空间清楚，而且既能服务真实业务，也能讲成课程案例。

## 成功标准

- 最小成功结果是什么：
  对 3 个真实竞品页面形成稳定监控，并输出一份可读的变化摘要。
- 用户拿到结果后，能直接做什么下一步动作：
  判断是否需要复查竞品价格、更新产品策略、调整销售说法或进入进一步研究。
- 什么情况下算“做出来了但还不能交付”：
  虽然能看到变化，但输出还停留在原始 diff，无法支持业务判断。

## 输入

- 最小输入：
  竞品名称、网址、页面名称、页面类型、关注字段、检查频率。
- 可选输入：
  国家市场、产品线、是否关注价格、是否关注 CTA、备注说明。
- 输入示例：
  见 `examples/minimal-monitoring-input.json`

## 输出

- 核心输出：
  竞品变化摘要，包含变化页面、变化类型、疑似业务含义、人工复核建议。
- 输出格式：Markdown / JSON
- 输出示例：
  见 `examples/minimal-monitoring-output.md`

## 核心规则与边界

- 必须遵守的业务规则：
  首版只监控少量高价值页面，不追求覆盖全站。
- 哪些信息只能写成推断：
  价格调整意图、促销策略、库存意图、页面改版目的。
- 哪些情况必须提示人工复核：
  价格变化、货币变化、疑似新产品上线、疑似促销结束、页面结构异常。
- 明确不做什么：
  不重写底层监控系统；不先做大规模批量部署；不自动代替人做竞品结论。

## 依赖与实现方式

- 需要的脚本 / 工具 / API：
  `changedetection.io`、少量本地配置文件、`scripts/build_change_summary.py` 摘要脚本。
- 是否需要联网：
  需要。
- 是面向 Codex、本地脚本，还是 OpenClaw / 龙虾优先：
  首版优先面向 Codex / 本地脚本。
- 是否需要 `for-openclaw/` 变体：
  首版不需要。

## 课程与内容映射

- 对应课程：
  AI 编程实操课第 9 课《竞品数据监控工具》。
- 可讲成什么案例：
  如何用成熟开源监控底座，快速搭出外贸竞品变化追踪最小版本。
- 可衍生什么内容选题：
  竞品价格怎么监控、上新怎么监控、哪些变化值得销售团队第一时间知道。

## 验收方式

- 最小测试输入是什么：
  3 个真实竞品页面，分别覆盖产品详情页、新品集合页、首页促销区。
- 预期输出应包含哪些关键信息：
  变化页面、变化类型、影响判断、人工复核建议。
- 如何判断这个 Skill 已达到可演示：
  能用固定样例讲清输入、变化、输出和业务解释。
- 如何判断这个 Skill 已达到可交付：
  能稳定接收少量真实页面变化，并输出业务可用摘要。
