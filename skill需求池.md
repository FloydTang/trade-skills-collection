# 外贸 Skill 需求池

这份文档用于统一维护外贸业务节点 Skill 的候选方向、状态和下一步动作。

使用规则：

- 新 Skill 必须先在这里立项，再进入开发
- 条目要写成业务化需求，不写成课程标题复述
- 每次状态变化，都要更新“当前状态”和“下一步动作”

## 与工作间的衔接规则

`skill需求池.md` 不是日常发现清单，也不是 GitHub 候选收集表。

GitHub 开源发现项目的日常沉淀，统一先进入：

- `工作间/开源工具总表.md`
- `工作间/路线映射.md`

只有满足以下条件，才允许从仓库内 `工作间/` 升级进入 `skill需求池.md`：

- 已经能明确对应哪条业务方向，而不是泛工具收藏
- 已经能明确更适合“独立新 Skill”还是“优化现有 Skill”
- 已经写清主要价值、主要边界和下一步动作
- 不是单日热点，至少具有持续复用或持续判断价值

补充规则：

- 进入 `skill需求池.md` 的稳定方向，后续应在 Obsidian `工具工作间/外贸skill` 有对应记录或说明，方便课程和产品表达使用

以下内容不要直接写进 `skill需求池.md`：

- 还没形成方向判断的观察项目
- 只有 Star 增速，没有业务挂靠关系的热门仓库
- 只是方法启发、但当前没有明确落地动作的候选

顶级原则保持一致：

`实事求是、简洁高效`

## 与工作间的衔接规则

`工作间/` 不是需求池，但它是当前 `外贸skill` 分类中，GitHub 开源发现项目进入需求池前的唯一长期判断入口。

它在仓库内只是当前分类的记录层，对应的是 Obsidian `工具工作间/外贸skill`，不等于 Obsidian 顶层“工具工作间”。

`skill需求池.md` 是仓库内的正式立项入口；一旦方向稳定，也应在 Obsidian 对应分类下可见。

升级规则只保留 4 条：

- 每日日报不能直接进入 `skill需求池.md`
- 只有已经进入 `工作间/开源工具总表.md` 的项目，才允许继续判断是否升级
- 只有在 `工作间/路线映射.md` 中已经写清“挂靠目标、落地方式、判断原因”的项目，才允许进入需求池
- 如果只是值得继续观察、但还说不清要独立做还是优化现有 Skill，就继续留在工作间，不升级

默认升级路径：

`GitHub开源发现自动化/reports/每日日报`
-> `工作间/开源工具总表.md`
-> `工作间/路线映射.md`
-> `skill需求池.md`

## 当前发布策略说明

这个母目录当前采用“三层结构”推进：

1. 单节点 Skill：当前主发布单位，也是课堂上即插即用的最小单元
2. 链路组合包：当多个节点稳定后，用于课程演示、业务闭环验证和轻量自动化复用
3. 总 Skill / 总编排器：放在后续阶段，负责调度已有节点 Skill，不进入当前单节点发布节奏

当前推荐主动开发链路为：

`客户搜索 / 线索发现 -> 线索整理 / 初筛 -> 客户背调 -> 开发信 -> 跟进优先级 / 客户管理`

当前优先策略：

- 先把单节点 Skill 做到可演示或可交付
- 再逐步组织链路组合包
- 总编排器作为后续阶段，不单独立项为当前主线条目
- 当前维护重点先锁定在 4 个已完成节点：`客户搜索skill`、`线索整理skill`、`客户背调skill`、`开发信skill`
- 其他目录和方向当前不作为优先安装、优先测试或优先修补对象

当前已进入设计准备的链路组合包方向：

- `客户搜索skill -> 线索整理skill -> 客户背调skill -> 开发信skill`
- 该方向暂定命名为：`主动开发最小闭环链路组合包`
- 定位：课程演示、会员权益交付、业务闭环验证

## 当前优先级说明

- 第一优先级：能直接服务“主动开发闭环”演示和培训增长的方向
- 第二优先级：能补齐业务闭环、但暂不急于开发的方向
- 被动获客相关方向暂不删除，但降为后续扩展
- 当前优先锁定首批 5 个主方向，避免范围失控

## 1. 客户搜索 / 线索发现 Skill

- Skill 名称：客户搜索 / 线索发现 Skill
- 所属业务节点：主动开发入口 / 客户搜索
- 用户痛点：知道大概要找哪类客户，但不会系统搜，不会快速从公开网页和 LinkedIn 找出第一批潜在客户
- 典型输入：目标市场、产品、客户类型、关键词、补充限制条件
- 目标输出：结构化候选客户名单，包含公司名、来源链接、可见联系人线索、补查建议
- 适用对象：外贸业务员、销售主管、老板、课程学员
- 业务价值：把“手工搜客户”升级成“可复制的客户发现流程”，帮助企业更快建立主动开发起点
- 对应课程/内容来源：外贸 AI 落地营中的主动开发样板流程、AI 搜客户方法表达
- 需要的能力模块：网页搜索、LinkedIn 搜索、结果去重、基础字段抽取、来源记录
- 风险与边界：只用公开信息；不承诺完整名单；不做大规模爬取；优先通用轻量路线
- 当前状态：可交付
- 下一步动作：保留为主动开发链路入口节点的单节点可交付版本；独立仓库对应 `https://github.com/FloydTang/trade-lead-discovery`；后续继续优化真实联网搜索源稳定性与结果过滤质量

## 2. 线索整理 / 初筛 Skill

- Skill 名称：线索整理 / 初筛 Skill
- 所属业务节点：主动开发中间层 / 名单整理
- 用户痛点：搜到一堆公司名、链接和零散联系人后，信息太乱，无法快速变成可判断、可跟进的线索池
- 典型输入：候选客户名单、公司链接、联系人线索、备注
- 目标输出：统一字段后的线索池，包含缺失项、初步分类、可补查建议
- 适用对象：外贸业务员、销售支持、课程学员
- 业务价值：把零散搜索结果整理成能进入背调和开发的标准输入，减少后续判断成本
- 对应课程/内容来源：外贸 AI 落地营中的线索整理 / 初筛逻辑
- 需要的能力模块：字段规范化、缺失检测、初步分类、优先级提示、导出能力
- 风险与边界：初筛只是辅助，不替代人工判断；不做复杂 CRM；首版以轻量结构为主
- 当前状态：可交付
- 下一步动作：保留为主动开发链路中的单节点可交付节点；独立仓库对应 `https://github.com/FloydTang/trade-lead-screening`；后续可继续补 CSV 导出和更细行业分类规则

## 3. 客户背调 Skill

- Skill 名称：客户背调 Skill
- 所属业务节点：主动开发判断中枢 / 客户开发前置研究
- 用户痛点：拿到公司名、邮箱或联系人后，不知道客户真假、官网真假、联系人真假，也无法快速整理成销售可用判断
- 典型输入：公司名、联系人名、邮箱、官网、国家市场、补充备注
- 目标输出：结构化客户情报报告，包含身份快照、数字足迹、风险评级、销售切入角度、可选英文触达草稿
- 适用对象：外贸业务员、销售支持、老板、课程学员
- 业务价值：降低主动开发中的盲目投入，把零散搜索升级成证据驱动判断，为首轮触达提供依据
- 对应课程/内容来源：外贸专区第1课《客户背调 Skill》、AI编程实操课第8课、第12课、外贸 AI 落地案例内容
- 需要的能力模块：网页搜索、页面抓取、实体匹配、证据分级、风险判断、报告生成
- 风险与边界：只使用公开信息；弱匹配必须标注不确定；不做法律或 KYC 级尽调；不捏造个性化内容
- 当前状态：可演示
- 下一步动作：保留为母目录样板 Skill；作为单节点可演示节点继续维护，并补充其在“搜索 / 初筛 / 开发信”组合链路中的衔接说明；独立仓库对应 `https://github.com/FloydTang/trade-customer-intel`，合集仓库保留稳定副本

## 4. 开发信 Skill

- Skill 名称：开发信 Skill
- 所属业务节点：主动开发触达 / 首轮开发 / 报价与跟进
- 用户痛点：开发信、报价邮件和跟进邮件长期依赖人工拼凑，效率低、针对性弱、风格不稳定
- 典型输入：客户基础信息、客户画像摘要、产品信息、报价范围、跟进阶段、目标语气
- 目标输出：可直接修改发送的英文开发信、报价邮件、跟进邮件，支持多个版本或风格
- 适用对象：外贸业务员、销售支持、课程学员
- 业务价值：缩短主动开发中的触达准备时间，提升触达一致性，把背调结果转成实际销售动作
- 对应课程/内容来源：AI编程实操课第10课《外贸邮件自动生成器》、外贸内容中关于客户开发与转化的话题
- 需要的能力模块：模板管理、变量填充、风格控制、版本生成、输入标准化
- 风险与边界：不捏造客户需求；不把未确认事实写进邮件；不直接代替人工审核和发送
- 当前状态：可交付
- 下一步动作：保留为主动开发链路中的单节点可交付节点，进入合集仓库增量推送，并按与客户背调 Skill 相同的双轨逻辑维护独立仓库；独立仓库对应 `https://github.com/FloydTang/trade-outreach-email`；后续围绕 `follow_up` 模板质量、CI 接入和 OpenClaw 实战反馈继续增强

## 5. 跟进优先级 / 客户管理 Skill

- Skill 名称：跟进优先级 / 客户管理 Skill
- 所属业务节点：主动开发后续承接 / CRM 轻量管理
- 用户痛点：主动开发出来的客户多了之后，不知道今天先跟谁，跟进记录又容易分散和漏掉
- 典型输入：客户基础资料、最近互动时间、风险等级、阶段标签、下次动作、备注
- 目标输出：按优先级排序的待跟进清单，或结构化客户档案与基础提醒结果
- 适用对象：外贸业务员、销售支持、小团队管理者、课程学员
- 业务价值：把主动开发出来的客户线索沉淀下来，减少漏跟进和低价值消耗
- 对应课程/内容来源：外贸真实业务动作、客户背调与开发信之后的承接需求、AI 工作流课程中的任务排序思路
- 需要的能力模块：状态更新、排序规则、搜索过滤、提醒逻辑、导出能力
- 风险与边界：默认轻量本地版本，不先做复杂数据库；排序只作辅助决策，不替代业务判断
- 当前状态：候选
- 下一步动作：先拆成“跟进优先级”与“客户管理”两条轻量路线，再决定首版先做哪个，并明确其作为主动开发链路后续承接节点的单节点发布方式

## 6. 主动开发最小闭环链路组合包

- Skill / 方案名称：主动开发最小闭环链路组合包
- 所属层级：链路组合包
- 所属业务链路：`客户搜索skill -> 线索整理skill -> 客户背调skill -> 开发信skill`
- 用户痛点：单个节点已经可用，但用户不知道如何把搜索、整理、背调和开发信串成一个可演示、可复用、可交付的最小主动开发闭环
- 典型输入：目标市场、产品、客户类型、关键词、补充限制条件，以及固定样例输入包
- 目标输出：一套从候选客户发现到英文开发信草稿的完整演示结果，包含每一步中间产物、桥接文件和人工复核点
- 适用对象：课程学员、需要快速理解主动开发工作流的外贸从业者、会员用户、内部演示人员
- 业务价值：把“多个单点 Skill”升级成“能讲、能跑、能交付的主动开发闭环样板”，便于课程表达、会员权益包装和后续轻量自动化
- 对应课程/内容来源：外贸 AI 落地营的主动开发主链路、单节点 Skill 的组合演示、会员权益中的部署包与案例包
- 需要的能力模块：现有四个节点 Skill 的桥接脚本复用、样例衔接、最小串联脚本、人工复核点说明、组合包 README
- 风险与边界：不重写现有单节点业务逻辑；不直接做重型总编排器；不默认自动发信；不把会员权益交付依赖在 GitHub 公开仓库本身
- 当前状态：设计中
- 下一步动作：使用 `模板/skill立项模板.md` 完成立项方案；建立组合包目录；补齐固定样例链路；准备一套适合 brand-site 展示和会员权益分发的说明文档

## 7. 外贸 Skill 开发超能力

- Skill / 方案名称：外贸 Skill 开发超能力
- 所属层级：内部方法层 / 研发与交付增强包
- 所属业务链路：服务 `客户搜索skill`、`线索整理skill`、`客户背调skill`、`开发信skill` 与 `主动开发链路组合包`

- 典型输入：业务问题描述、拟做的 Skill 方向、现有样例、上下游衔接关系、当前状态与交付目标
- 目标输出：面向内部研发与交付的结构化方法产物，包含业务拆解、立项方案、验收判断、链路设计卡、发布包装建议
- 适用对象：半斤九两科技内部研发人员、课程研发人员、需要继续扩展外贸 Skill 的协作者
- 业务价值：把“如何做出一个合格 Skill”这件事也做成可复用方法资产，减少重复试错，让单节点、组合包和课程交付之间的衔接更稳定
- 对应课程/内容来源：AI 编程实操课中的 Skill 开发方法表达、外贸 AI 落地营中的节点设计与链路演示经验、当前母目录的模板与双轨分发策略
- 需要的能力模块：业务拆解、Skill 立项、验收判断、链路串联、发布包装、人工复核点设计
- 风险与边界：只服务内部方法沉淀，不替代现有业务节点 Skill；不原样照搬重工程流程；不把通用软件工程纪律强塞给课程学员或终端客户
- 当前状态：设计中
- 下一步动作：先完成立项方案，明确课程版与客户版口径差异；随后仅作为内部方法层试运行，优先服务 `客户搜索skill`、`客户背调skill` 与 `开发信skill` 的后续迭代

## 8. 竞品监控 Skill

- Skill 名称：竞品监控 Skill
- 所属业务节点：市场研究 / 竞品跟踪
- 用户痛点：竞品官网、价格、新品和动态需要反复人工查看，耗时且难持续
- 典型输入：竞品网址列表、监控频率、关注字段、通知方式
- 目标输出：竞品变化汇总报告，突出价格变化、上新动态、促销活动和重点异常
- 适用对象：老板、运营、销售支持、市场研究人员、课程学员
- 业务价值：提高市场反应速度，帮助及时调整报价、选品和客户沟通策略
- 对应课程/内容来源：AI编程实操课第9课《竞品数据监控工具》、AI+外贸研究和市场洞察内容
- 需要的能力模块：定时抓取、结构化抽取、历史对比、异常标注、报告通知
- 风险与边界：遵守目标网站访问边界；不承诺全站稳定抓取；网站结构变化需要维护
- 当前状态：开发中
- 下一步动作：首版继续采用 `changedetection.io` 作为上游监控底座；当前已锁定 Allbirds 的 3 个真实公开页面（产品详情页、新品集合页、首页促销区），并补齐最小变化摘要脚本与样例输入输出；下一步是在真实 watch 上验证一次“变化抓取 -> 变化摘要 -> 人工复核建议”链路，确认字段选择器是否稳定；同时把这条方向的长期说明同步保持在 `工作间/` 与 Obsidian `工具工作间/外贸skill` 可见
