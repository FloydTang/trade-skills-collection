# 外贸 Skill 开发超能力立项方案

## 基本信息

- Skill 名称：外贸 Skill 开发超能力
- 所属业务节点：内部方法层 / 外贸 Skill 研发与交付增强
- 负责人：Codex 协作起草，后续由半斤九两科技继续推进
- 当前状态：设计中

## 业务问题

- 这个 Skill 要解决的具体问题是什么：
  当前母目录已经有多个可演示或可交付节点，也有主动开发链路组合包，但在新增方向立项、输入输出收敛、边界说明、验收补齐、链路串联和对外包装时，仍然高度依赖个人经验，缺少一套稳定的内部方法层。
- 当前人工是怎么做的：
  通常由负责人临时判断业务问题怎么拆、模板怎么填、目录怎么建、样例怎么补、何时可演示或可交付，再在 README、立项方案、验收清单和组合包说明之间手工来回对齐。
- 为什么值得被做成 Skill：
  它不新增某个业务节点功能，但能提升整个母目录继续扩展时的稳定性，降低重复试错成本，也更利于课程表达、会员承接和后续定制交付。

## 成功标准

- 最小成功结果是什么：
  针对一个新的外贸 Skill 方向或一个已有节点的增强任务，能够稳定产出一套结构化方法产物，包括业务拆解、立项方案、验收判断、链路位置和发布包装建议。
- 用户拿到结果后，能直接做什么下一步动作：
  直接进入单节点 Skill 开发，或决定是否进入组合包演示、合集收录、课程案例制作与会员交付说明整理。
- 什么情况下算“做出来了但还不能交付”：
  只有概念性方法说明，没有对接当前母目录模板、没有示例、没有明确边界、也不能映射到单节点 / 组合包发布节奏时。

## 输入

- 最小输入：
  - `business_problem`
  - `target_skill_name`
  - `current_stage`
  - `expected_outcome`
- 可选输入：
  - `upstream_or_downstream_skills`
  - `sample_input_or_fixture`
  - `delivery_context`
  - `course_context`
  - `notes`
- 输入示例：

```json
{
  "business_problem": "新增一个外贸展会线索筛选方向，但还没有收敛最小输入输出和是否应先做单节点版本。",
  "target_skill_name": "展会线索筛选 Skill",
  "current_stage": "候选",
  "expected_outcome": "形成可进入开发的立项方案，并判断它和主动开发链路的关系。",
  "upstream_or_downstream_skills": ["客户搜索skill", "线索整理skill"],
  "delivery_context": "希望后续能作为课程案例和会员补充包",
  "course_context": "外贸 AI 落地营"
}
```

## 输出

- 核心输出：
  - 业务拆解结果
  - Skill 立项草案
  - 验收与风险判断
  - 链路位置建议
  - 发布包装建议
- 输出格式：Markdown 为主，必要时补 JSON 结构卡片
- 输出示例：

```text
方向判断：建议先做单节点 Skill，不直接进入总编排器

最小输入：
- 展会名称
- 线索来源表
- 目标产品关键词

核心输出：
- 线索分桶结果
- 建议进入客户背调的候选名单
- 人工复核项

验收建议：
- 至少 1 份固定样例
- 明确与线索整理 Skill 的衔接字段
- 补清楚哪些判断只能写成推断
```

## 核心规则与边界

- 必须遵守的业务规则：
  - 先服务当前母目录已有发布结构，不另起一套方法论体系
  - 优先保证单节点 Skill 可独立成立，再讨论链路组合和总编排器
  - 默认对接现有 `模板/skill立项模板.md`、`模板/skill验收清单.md` 和合集收录标准
- 哪些信息只能写成推断：
  - 用户真实购买意愿
  - 市场优先级判断
  - 客户对课程或会员内容的付费倾向
  - 某个方向未来商业化价值的确定结论
- 哪些情况必须提示人工复核：
  - 方向本身是否属于真实业务节点
  - 是否值得单独成 Skill，还是只是现有 Skill 的增强项
  - 是否应进入公开分发、课程演示或会员承接
  - 是否会让内部方法层过度工程化
- 明确不做什么：
  - 不替代现有业务节点 Skill 的实际执行逻辑
  - 不原样照搬通用软件工程重流程
  - 不直接作为面向客户的产品卖点
  - 不跳过现有立项与验收规则

  - 首版优先面向 Codex / 内部本地协作
- 是否需要 `for-openclaw/` 变体：
  - 首版不需要

## 课程与内容映射

- 对应课程：
  - AI 编程实操课中的 Skill 研发方法表达
  - 外贸 AI 落地营中的“从业务问题到可交付 Skill”内容
- 可讲成什么案例：
  - 为什么先做单节点，再做组合包
  - 为什么 Skill 的输入、输出、边界和样例必须一起设计
  - 如何把一个业务方向从“想法”变成“可演示”
- 可衍生什么内容选题：
  - 外贸 Skill 为什么不能只靠提示词拼凑
  - 如何判断一个方向该做成单节点 Skill，还是该做成组合包
  - 如何把 AI 工具包装成可交付业务资产

## 验收方式

- 最小测试输入是什么：
  - 至少 1 个新方向立项需求
  - 至少 1 个已有节点 Skill 的增强需求
- 预期输出应包含哪些关键信息：
  - 业务问题定义
  - 最小输入与明确输出
  - 风险与边界
  - 链路位置
  - 验收与发布建议
- 如何判断这个 Skill 已达到可演示：
  - 能稳定把一个真实方向整理成完整立项方案，并能映射到当前母目录模板和发布节奏。
- 如何判断这个 Skill 已达到可交付：
  - 有固定方法说明、最小案例、清楚的课程版与客户版口径区分，并已在至少 2 个真实方向上验证可复用。
