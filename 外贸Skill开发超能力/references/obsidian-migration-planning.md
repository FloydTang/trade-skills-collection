# Obsidian 内容库与工作台整合迁移规划参考

这个参考文件只服务一类任务：

- 把多个内容库、资料库、工作台整合成一套更稳定的 Obsidian 工作系统

目标不是先迁完，而是先迁对。

## 一、先判断当前整合对象

至少先列清这些对象：

- 现有有哪些 Vault 或内容库
- 哪些是长期资料库
- 哪些是当前高频工作台
- 哪些是课程内容库
- 哪些是客户交付资料
- 哪些是临时缓存或待整理区

如果这些对象都还没列清，不要直接进入目录设计。

## 二、建议先分 4 层

### 1. 工作台层

放每天高频操作的页面和面板，例如：

- 今日工作台
- 客户推进面板
- 内容发布面板
- 课程研发面板
- Skill 孵化面板

特点：

- 高频打开
- 以行动为主
- 不承载大量原始资料

### 2. 项目层

放有明确起止、负责人和产出的事项，例如：

- 某门课程
- 某个客户交付
- 某个 Skill 开发
- 某次迁移计划

特点：

- 有阶段
- 有产出
- 有验收

### 3. 资料库层

放长期积累但不需要每天操作的材料，例如：

- 课程素材
- 行业研究
- 业务 SOP
- 模板库
- 案例库

特点：

- 重检索
- 重引用
- 轻编辑

### 4. 输出层

放已经准备对外或对内交付的内容，例如：

- 文章成稿
- 课程讲义定稿
- 客户方案定稿
- Skill 对外说明

特点：

- 结构相对稳定
- 命名需要清楚
- 便于导出和复用

## 三、迁移顺序建议

默认建议按这个顺序：

1. 先设计目标结构
2. 再圈定高频工作台
3. 再迁当前活跃项目
4. 再迁高复用资料库
5. 最后处理历史归档与低频内容

不要反过来从历史旧资料开始清。

## 四、迁移时必须单独检查的风险

### 1. 链接风险

- 双链是否会失效
- MOC 或索引页是否依赖旧路径
- 嵌入块是否依赖旧标题

### 2. 附件风险

- 图片、PDF、音频是否分散在多个 Vault
- 是否存在重名附件
- 是否有引用路径写死

### 3. 模板风险

- 模板调用目录是否变化
- Dataview/Templater 查询是否依赖旧路径
- 快捷工作流是否会因目录变化失效

### 4. 命名风险

- 是否同时存在“业务命名”和“内容命名”两套标准
- 是否为了统一命名而破坏历史链接

## 五、迁移计划最小输出格式

```text
一、当前内容对象清单
二、目标 Vault 结构
三、第一阶段迁移范围
四、暂不迁移范围
五、人工确认点
六、验收标准
```

## 六、常见错误

- 先搬文件，后想结构
- 先讨论插件，后讨论工作流
- 想一次性完成全部整理
- 把资料库和工作台混成一个层级
- 没有定义“暂不迁移”的范围

## 七、适合进一步拆成 Skill 的方向

如果迁移规划稳定，后续可以继续拆成：

- Obsidian 内容分层 Skill
- Obsidian 工作台设计 Skill
- 内容库迁移验收 Skill
- 知识资产归档 Skill
