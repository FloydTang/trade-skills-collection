# 开发信 Skill 立项方案

## 基本信息

- Skill 名称：开发信 Skill
- 所属业务节点：客户触达 / 首轮开发 / 跟进转化
- 负责人：Codex 协作草案
- 当前状态：设计中

## 业务问题

- 这个 Skill 要解决的具体问题是什么：
  外贸业务员在首轮触达和后续跟进时，常常需要根据零散客户信息临时拼邮件，导致写作慢、风格不稳定、信息复用差，也很难把前置背调结果真正转成销售动作。
- 当前人工是怎么做的：
  业务员通常复制历史邮件模板，再手动替换公司名、产品名、国家市场和少量客户背景信息；如果没有现成模板，就临时写一版，再靠个人经验调整语气和结构。
- 为什么值得被做成 Skill：
  这是高频、重复、可结构化的业务动作，而且和客户背调 Skill 天然衔接。只要把输入范围收敛清楚，就能快速形成“背调 -> 开发信/跟进信”的最小业务闭环。

## 成功标准

- 最小成功结果是什么：
  用户输入最少的客户信息、产品信息和跟进阶段后，Skill 能稳定输出 1 封首轮开发信或 1 封跟进邮件草稿，且包含明确主题、正文结构、可人工修改的 CTA。
- 用户拿到结果后，能直接做什么下一步动作：
  人工快速审阅，补充价格、交期、附件等实际信息后发送给客户，或继续复制为 CRM/邮件系统中的标准草稿。
- 什么情况下算“做出来了但还不能交付”：
  只能偶尔生成可用文案，但输入要求不清楚、容易捏造客户信息、风格不可控，或无法解释哪些内容必须人工复核。

## 输入

- 最小输入：
  - `email_type`：`first_touch` 或 `follow_up`
  - `customer_name`
  - `company_name`
  - `product_or_offer`
  - `goal`
- 可选输入：
  - `country_or_market`
  - `customer_profile_summary`
  - `previous_contact_context`
  - `tone`
  - `sender_name`
  - `sender_company`
  - `signature`
  - `constraints`
- 输入示例：

```json
{
  "email_type": "first_touch",
  "customer_name": "Anna",
  "company_name": "Acme Foods",
  "product_or_offer": "frozen mixed vegetables",
  "goal": "introduce our factory and ask whether they are open to new suppliers",
  "country_or_market": "Poland",
  "customer_profile_summary": "Company website shows private-label frozen food focus in EU retail.",
  "tone": "professional,warm",
  "sender_name": "Leo",
  "sender_company": "Ningbo FreshGrow Foods"
}
```

## 输出

- 核心输出：
  - 邮件使用建议
  - 英文邮件标题候选 2 个
  - 英文正文草稿 1-2 个版本
  - 中文复核提示
  - 使用到的输入要点列表
- 输出格式：Markdown + 可选 JSON
- 输出示例：

```text
Email Type: First Touch

Subject Options
1. Frozen Mixed Vegetables Supply for Acme Foods
2. Possible Private-Label Frozen Vegetable Support for Your EU Lines

Draft Email
Dear Anna,
...

Review Notes
- “private-label” based on provided customer profile summary; verify before sending.
- No pricing or MOQ included. Add only if confirmed internally.
```

## 核心规则与边界

- 必须遵守的业务规则：
  - 只使用用户明确提供的信息和已确认的客户画像摘要。
  - 邮件目标必须和 `email_type` 一致，不能把首轮开发信写成报价邮件。
  - 输出以“可人工修改后发送”为目标，不默认直接可发。
- 哪些信息只能写成推断：
  - 客户采购偏好、产品兴趣点、渠道类型、市场策略等，除非输入已明确说明，否则只能写成保守表达或不写。
- 哪些情况必须提示人工复核：
  - 使用了客户画像摘要中的行业、渠道、市场定位信息。
  - 提到价格、MOQ、交期、认证、合作案例、交付能力等易引发承诺的信息。
  - 跟进邮件中引用了历史沟通，但输入里的历史上下文不完整。
- 明确不做什么：
  - 不自动发送邮件。
  - 不生成“强个性化”但无证据支撑的句子。
  - 不在首版中覆盖报价邮件、催款邮件、投诉回复等复杂场景。

## 依赖与实现方式

- 需要的脚本 / 工具 / API：
  - 首版可先做本地模板编排与规则生成，不强依赖外部 API。
  - 如需后续增强，可接入客户背调 Skill 输出作为上游输入。
- 是否需要联网：
  - 首版不强制联网。
- 是面向 Codex、本地脚本，还是 OpenClaw / 龙虾优先：
  - 首版优先面向 Codex / 本地脚本。
- 是否需要 `for-openclaw/` 变体：
  - 首版不需要，待输入结构稳定后再评估。

## 课程与内容映射

- 对应课程：
  - AI 编程实操课第 10 课《外贸邮件自动生成器》
  - 可与客户背调 Skill 串联成“从背调到触达”的连续案例
- 可讲成什么案例：
  - 如何把客户背调摘要转成首轮开发信
  - 如何把跟进阶段和历史沟通转成更稳妥的英文跟进邮件
- 可衍生什么内容选题：
  - 外贸开发信为什么不能只靠万能模板
  - 哪些客户信息适合写进首轮邮件，哪些必须谨慎
  - 背调结果如何真正接入销售动作

## 验收方式

- 最小测试输入是什么：
  - 1 个 `first_touch` 样例
  - 1 个 `follow_up` 样例
- 预期输出应包含哪些关键信息：
  - 明确邮件类型
  - 2 个标题候选
  - 1-2 个英文邮件版本
  - 中文复核提示
  - 输入依据列表或关键变量回显
- 如何判断这个 Skill 已达到可演示：
  - 能用固定样例稳定生成结构一致、语气可读、无明显捏造的邮件草稿。
- 如何判断这个 Skill 已达到可交付：
  - 输入格式固定，输出结构固定，边界说明清楚，并有最小示例和脚本入口可重复运行。
