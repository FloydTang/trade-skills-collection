# 飞书留痕字段映射

这份说明只服务一个目标：

- 把当前 4 个已跑通节点的阶段产物，稳定映射成 OpenClaw 可落到飞书云表格 / 云文档的统一结构

当前默认入口：

- 先跑 `主动开发链路组合包/scripts/run_minimal_demo.py`
- 然后读取输出目录里的 `09-feishu-workflow-bundle.json`

这个 bundle 不直接调用飞书 API。

它的职责是：

- 给 OpenClaw 一份稳定的写库协议
- 明确每个阶段应该创建什么资产
- 明确主表回写规则和重跑策略

配套执行顺序见：

- `references/OpenClaw执行规范.md`

## 飞书容器结构

当前推荐结构不是“每个阶段创建一个新的多维表格”。

当前默认结构是：

- 只使用 1 个主 Base
- 在同一个 Base 下维护多张子表
- 在同一个工作空间命名体系下维护客户背调和开发信文档

推荐主 Base 名称：

- `Trade Lead Workflow Hub`

同一个 Base 下默认维护：

- `Lead Workflow Master`
- `Lead Discovery Results`
- `Lead Screening Results`

固定规则：

1. 先查主 Base，存在则复用
2. 缺子表时只补子表，不新建第二个 Base
3. 单点运行也必须挂到同一个主 Base 下

## 总体结构

`09-feishu-workflow-bundle.json` 当前至少包含 6 类核心信息：

- `workflow_meta`
- `workspace_container`
- `master_table_schema`
- `stage_assets`
- `master_records`
- `openclaw_handoff`

其中：

- `workspace_container` 用来描述“单 Base 多子表”的飞书容器规则
- `master_table_schema` 用来创建或校验 `Lead Workflow Master`
- `stage_assets` 用来指导每个步骤应该创建表记录还是文档
- `master_records` 用来在每次运行后回写主表当前状态
- `openclaw_handoff` 用来描述执行顺序、单点接入、失败处理和重跑策略

## 主表字段

主表固定名称：

- `Lead Workflow Master`

首期固定字段：

- `workflow_id`
- `lead_id`
- `company_name`
- `person_name`
- `email`
- `country_or_market`
- `source_stage`
- `current_stage`
- `current_status`
- `recommended_next_action`
- `risk_rating`
- `entity_confidence`
- `owner`
- `next_follow_up_date`
- `last_updated_at`
- `search_asset_ref`
- `screening_asset_ref`
- `intel_asset_ref`
- `email_asset_ref`
- `combo_run_id`
- `asset_keys`
- `failure_reason`

补充说明：

- `*_asset_ref` 默认写文本引用，不强依赖飞书 URL 字段
- 文本引用里可以存 record URL、doc URL、筛选视图链接或 asset token
- `asset_keys` 用来保存 OpenClaw 创建资产前后的稳定映射键
- `failure_reason` 只在某一步执行失败时回写
- 首次试跑优先保证“写得进去”，不要把 URL 字段当成硬前置

## 单点接入主表规则

当前固定要求：

1. 单独跑客户背调时，也必须先查 `Lead Workflow Master`
2. 单独跑开发信时，也必须先查 `Lead Workflow Master`
3. 同一 lead 补跑其他节点时，优先更新原主记录
4. 不新建平行 lead

推荐匹配顺序：

1. 先按 `lead_id`
2. 再按 `company_name + company_website/domain + email`

如果仍无法确认是同一主体：

- 主表写 `failure_reason` 或人工复核原因
- 本轮不自动创建下游文档

文档正文建议直接按 `document_blocks` 渲染，重点保留：

- Scenario
- Subject Options
- Draft Version A / B
- Review Notes
- Input Signals Used

## 回写规则

固定规则如下：

1. 先查主 Base，再查固定子表，不额外新建新的多维表格
2. 搜索完成后，先写 `Lead Discovery Results`
3. 初筛完成后，写 `Lead Screening Results`
4. 再根据 `master_records` 回写主表
5. 仅当 `recommended_next_action=enter_customer_intel` 且该 lead 被本轮选中时，创建客户背调文档
6. 客户背调完成后，再创建开发信文档
7. 开发信完成后，主表把该 lead 更新为 `current_stage=outreach_email`、`current_status=draft_ready`

补充说明：

- 主表是唯一总索引，不论全链路还是单点运行都必须接入
- 失败时也要优先回写主表状态，不因为单个 URL 或单个文档步骤失败而放弃整轮留痕

## 重跑策略

首期默认策略：

- 表格记录：按 `combo_run_id + lead_id` 做 upsert
- 背调文档：优先复用旧文档，再追加一个新版本区块
- 开发信文档：优先复用旧文档，再追加一个新草稿版本
- 主表记录：优先按 `lead_id` 复用，找不到再按公司名、域名、邮箱组合判断
- 失败补跑：从原主记录继续，保留已成功资产，不另起一条平行记录

这样做的原因：

- 避免飞书里反复新建一堆重复资产
- 避免每个阶段分散到不同 Base
- 保留历史版本
- 方便后续持续跟进和状态更新
- 更适合多人协作、仪表盘统计和长期扩展
