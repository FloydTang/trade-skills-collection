# 课程演示路径

## 目标

这套链路组合包在课程里不是为了展示“自动化很炫”，而是为了让学员理解：

- 为什么单节点 Skill 先行
- 为什么链路组合包要保留中间产物
- 为什么主动开发不能直接跳到一键全自动

## 推荐演示顺序

### 第一段：先讲方法论

建议先讲清三层结构：

1. 单节点 Skill
2. 链路组合包
3. 总 Skill / 总编排器

需要强调：

- 当前演示的是第二层
- 第二层的目标是闭环验证、课程表达和交付复用
- 不是为了取代四个单节点

### 第二段：讲四个节点各自解决什么问题

建议按业务动作讲，而不是按仓库讲：

1. 搜客户
2. 整理线索
3. 判断值不值得开发
4. 生成首轮触达草稿

### 第三段：讲为什么必须保留中间产物

这里建议重点讲三件事：

- 搜索结果不能直接当事实
- 线索整理结果决定了谁能进入背调
- 背调报告和开发信都必须留人工复核点

### 第四段：跑固定样例

推荐操作顺序：

1. 先展示 `examples/fixed-search-brief.json`
2. 解释搜索阶段为什么先用固定输出
3. 运行组合包串联脚本
4. 打开每个阶段的输出文件
5. 停在 `05-selected-customer-intel-input.json` 解释人工选择
6. 展示固定背调报告 fixture
7. 展示最终邮件草稿并指出必须人工改写的部分

### 第五段：讲“可演示”和“可交付”区别

建议说明：

- 可演示：能讲、能跑、能看到中间产物
- 可交付：除了能演示，还能让会员拿回去自己跑

## 演示时重点提醒

- 不把样例中的推断说成事实
- 不把邮件草稿说成“可直接发”
- 不把固定样例说成实时抓取结果
- 不把组合包说成总编排器

## 推荐收尾话术方向

建议最后落回业务价值：

- 不是多了一个花哨系统
- 而是把四个已经能用的节点，组织成了一个真实业务闭环样板
- 学员和会员拿到后，知道从哪里替换成自己的行业样例
