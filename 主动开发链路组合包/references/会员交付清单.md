# 公开分发与后续承接清单

## 这份清单解决什么问题

它用于回答一个很实际的问题：

- 如果把这个组合包直接公开出去，首版到底应该放什么
- 哪些内容必须带上
- 哪些内容当前先不承诺

## 首版建议公开内容

### 一、必须包含

- `README.md`
- `立项方案.md`
- `examples/fixed-search-brief.json`
- `examples/reviewed-customer-intel-report.json`
- `references/固定样例方案.md`
- `references/课程演示路径.md`
- `references/运行与安装.md`
- `references/会员交付清单.md`
- `references/验收与边界.md`
- `references/目录与命名约定.md`
- `scripts/run_minimal_demo.py`

### 二、建议一并带上

- 一份已跑好的 `outputs/demo-run/`
- 当前四个节点 Skill 的目录说明或对应仓库链接
- 一页面向新用户的“先看什么”说明
- 一份“飞书增强入口”说明，明确告诉用户这里可以复制增强执行词给龙虾

### 三、当前不建议直接打进首版公开包

- brand-site 前台页面文件
- 自动会员鉴权逻辑
- 自动下载发放逻辑
- 总编排器相关设想
- 尚未稳定的实时联网演示结果

## 当前推荐公开形式

### 形式 A：公开 GitHub 仓库

适合：

- 想持续更新的用户
- 会使用 GitHub 的用户
- 需要继续二次改造的用户

### 形式 B：公开下载包 / ZIP

适合：

- 只想先拿到能跑通样例的用户
- 课程型和执行型用户
- 不想碰 GitHub 的会员

建议 ZIP 内结构：

- 组合包目录
- 四个单节点的必要说明
- 一份欢迎说明
- 一份更新说明

## 建议附带的分发说明

公开时建议同步说明以下几点：

- 这是链路组合包，不是总编排器
- 默认以固定样例保证可演示
- 真实业务替换时，先换搜索输入和产品信息，不要一开始就改整个链路
- 邮件草稿必须人工复核后再用
- 飞书增强入口默认提供可直接复制给龙虾的增强执行词，而不是普通教程

## 用户拿到后最先应该做什么

建议引导用户按这个顺序：

1. 先读 `README.md`
2. 再读 `references/运行与安装.md`
3. 跑一次 `scripts/run_minimal_demo.py`
4. 看 `outputs/demo-run/` 里的 4 个关键文件
5. 再决定替换自己的行业样例

## 当前不应对外承诺的话

- “这个包已经全自动”
- “这个包能自动发信”
- “这个包能稳定抓取所有实时结果”
- “这个包已经等同于 CRM 或销售系统”

## 当前推荐对外口径

更稳的口径应是：

- 这是一个可跑通的主动开发最小闭环样板
- 它重点解决链路理解、样例复用、脚本串联和人工复核点保留
- 它适合公开安装、课程演示、内部培训和后续扩展起步
