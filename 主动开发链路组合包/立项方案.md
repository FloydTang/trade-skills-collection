# 主动开发最小闭环链路组合包立项方案

## 基本信息

- Skill 名称：主动开发最小闭环链路组合包
- 所属业务节点：主动开发链路组合 / 课程演示 / 会员交付
- 负责人：半斤九两科技
- 当前状态：设计中，已完成固定样例版最小演示骨架

## 业务问题

- 这个组合包要解决的具体问题是什么：
  当前 4 个节点已经分别可用，但用户很难直接理解如何把它们串成“可执行的主动开发闭环”。单节点能解决局部动作，不能直接展示完整业务路径。
- 当前人工是怎么做的：
  先人工搜索客户，再人工整理字段，再手工补背景信息，最后再写开发信。即使已有单节点脚本，很多人仍然不知道先跑哪个、输出给谁、哪里需要停下来复核。
- 为什么值得被做成链路组合包：
  因为它能把单点能力升级成业务闭环样板，既能用于课程讲解，也能作为会员权益中的“部署包”交付，还能为后续轻量自动化提供稳定基线。

## 成功标准

- 最小成功结果是什么：
  用一套固定样例，从客户搜索出发，稳定生成候选名单、线索整理结果、客户背调报告和首轮英文开发信草稿。
- 用户拿到结果后，能直接做什么下一步动作：
  可以理解整条主动开发链路；可以替换成自己的行业样例继续试跑；可以把结果用于课程演示、内部培训或会员交付说明。
- 什么情况下算“做出来了但还不能交付”：
  只有串联说明，没有固定样例；或能跑通但没有中间产物和人工复核点；或强依赖临时联网结果、无法稳定演示。

## 输入

- 最小输入：
  一套固定样例搜索条件，例如目标市场、产品、客户类型、关键词。
- 可选输入：
  限制条件、排除条件、行业备注、目标客户规模、目标国家等。
- 输入示例：
  参考 `客户搜索skill/examples/frozen-food-search.json` 这类固定样例，后续在本组合包目录中保留一份演示用输入副本或引用说明。

## 输出

- 核心输出：
  - 候选客户名单
  - 线索整理报告
  - 客户背调输入包
  - 客户情报报告
  - 开发信输入包
  - 英文开发信草稿
- 输出格式：Markdown / JSON
- 输出示例：
  以每一步的中间结果文件和最终开发信草稿为主，适合课程演示与会员部署包复核。

## 核心规则与边界

- 必须遵守的业务规则：
  - 只复用现有 4 个节点 Skill 的稳定能力
  - 中间产物必须可见、可保存、可人工复核
  - 不把推断写成确定事实
  - 不把“自动生成草稿”误写成“可直接自动发送”
- 哪些信息只能写成推断：
  - 搜索阶段的弱匹配客户
  - 背调阶段的弱证据身份判断
  - 开发信中的兴趣点、需求点、角色判断
- 哪些情况必须提示人工复核：
  - 上游候选客户证据弱
  - 关键字段缺失
  - 背调报告风险偏高或证据不足
  - 邮件内容涉及未确认的客户事实
- 明确不做什么：
  - 不重写现有单节点实现
  - 不直接做总编排器
  - 不做自动发信
  - 不做重型会员后台

## 课程与内容映射

- 对应课程：
  外贸 AI 落地营中的主动开发主链路演示、单节点 Skill 组合讲解、会员权益中的部署包说明。
- 可讲成什么案例：
  - 从搜客户到发出第一封开发信的最小闭环
  - 为什么要先做单节点，再做链路组合包
  - 如何保留人工复核点，而不是盲目全自动
- 可衍生什么内容选题：
  - 外贸主动开发的 AI 闭环怎么搭
  - 单个 Skill 和链路组合包有什么区别
  - 会员为什么拿到的是“可部署链路包”，而不是一堆零散脚本

## 验收方式

- 最小测试输入是什么：
  一套固定搜索样例，加上一条能够稳定串到开发信的演示链路。
- 预期输出应包含哪些关键信息：
  - 每一步清晰的输入输出文件
  - 线索整理后的标准字段
  - 客户背调报告中的证据与风险提示
  - 可人工修改的英文开发信草稿
- 如何判断这个组合包已达到可演示：
  文档能讲清；固定样例能跑通；中间产物齐全；人工复核点明确。
- 如何判断这个组合包已达到可交付：
  除了可演示条件外，还应具备清晰安装说明、稳定样例、可重复运行的最小串联脚本，以及面向会员交付的权益说明。
