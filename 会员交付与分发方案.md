# 外贸 Skill 公开分发与后续会员承接方案

## 这份方案解决什么问题

当前 4 个单节点 Skill 已经具备公开展示价值，而链路组合包现在也需要优先以公开方式分发，让更多用户和学员先装起来、跑起来、参与改进。

这份方案的目标不是做支付系统，而是明确：

- 哪些内容继续公开
- 哪些内容当前继续公开
- 哪些内容后续可以再进入会员承接
- `brand-site` 在整个体系中承担什么角色
- 会员交付首版应该怎么做

## 当前判断

当前阶段的优先策略已经调整为：

- 先公开
- 先让更多人跑起来
- 先降低安装门槛
- 先让学员和 agent 用户形成真实反馈

这意味着当前不再把“私有仓库 / 闭源分发”作为主策略，而是采用：

- 单节点 Skill 继续公开
- 链路组合包也进入公开仓库
- 会员承接先保留为后续增值层，而不是当前分发前提

基于当前本地 `brand-site` 目录形态，它适合做：

- 会员权益前台
- 工具分类目录
- 公开 / 会员内容说明页
- 联系入口与申请入口
- 公开工具的统一展示页

它当前不适合直接做：

- 下单系统
- 完整会员登录体系
- 受控文件下载后台
- 自动发放权限后台

因此现阶段应采用：

- `brand-site` 负责展示、导航、说明、联系承接
- GitHub 公开仓库负责当前主分发
- 会员承接先聚焦课程、答疑、案例和后续扩展包，而不是先做闭源资源隔离

## 分发结构建议

### 一、公开层

公开层继续承担内容传播、课程引流和开源信任建立作用。

建议继续公开：

- `客户搜索skill`
- `线索整理skill`
- `客户背调skill`
- `开发信skill`

公开层页面内容建议包括：

- 业务问题
- 输入输出说明
- 开源仓库链接
- 使用截图或样例结果
- 适合谁
- 当前状态
- 与上下游链路的关系

### 二、后续会员承接层

后续会员层不以“代码是否可下载”作为主要卖点，而以“更完整的案例、答疑、扩展包和课程承接”作为核心权益。

这里还应再补一条固定理解：

- 会员增强价值应优先体现为“高质量 agent 可执行资产”
- 例如：更完整的案例、部署辅导、飞书增强入口与配套服务

当前适合先公开：

- 主动开发最小闭环链路组合包
- 统一安装说明
- 串联脚本
- 固定演示样例

后续再考虑进入会员承接的内容：

- 更完整的行业案例包
- 后续链路扩展包
- 课程配套讲解材料
- 专门面向会员整理的部署辅导内容
- 会员专属服务说明与使用承接

### 三、承接层

`brand-site` 作为统一承接层，负责：

- 展示工具地图
- 标记哪些已经公开可安装
- 标记哪些后续会有扩展承接
- 解释“agent 友好型 Skill 部署包”是什么意思
- 把用户引导到 GitHub、课程介绍和后续联系入口

## brand-site 的推荐信息架构

建议新增一个“工具 / Skill”主入口，下面拆成 3 层。

### 1. 工具总览页

建议展示：

- 按业务链路分类，而不是只按仓库分类
- 每个工具的状态：已公开 / 后续扩展
- 每个工具的类型：单节点 Skill / 链路组合包
- 每个工具的定位：搜索、整理、背调、开发信、后续跟进

建议标签体系：

- `公开`
- `公开可安装`
- `单节点 Skill`
- `链路组合包`
- `Agent 友好型`
- `可部署`
- `可演示`

### 2. 单个工具详情页

公开工具详情页建议包含：

- 工具解决什么业务问题
- 输入与输出
- 最小使用方式
- GitHub 开源链接
- 上下游链路关系
- 如果想直接拿完整闭环，提示升级到会员权益

后续扩展页建议包含：

- 工具解决什么链路问题
- 包含哪些内容
- 当前哪些内容已公开
- 后续还会补什么
- 适合谁
- 如何继续深入使用
- 课程或联系入口

### 3. 后续承接页

后续承接页不需要做在线下单，但要把“为什么先开源、后做扩展承接”说清楚。

建议包含：

- 当前已经公开了什么
- 为什么现在优先公开安装与反馈闭环
- 后续扩展层会补什么
- 当前推荐的使用路径
- GitHub 与课程入口

## Agent 友好型 Skill 部署包的表达建议

站点上建议统一把这类产品定义成：

`Agent 友好型 Skill 部署包`

推荐解释口径：

- 不是空泛提示词包
- 不是只能看不能用的概念文档
- 而是围绕真实业务节点整理好的输入、输出、脚本、样例和说明
- 既适合人直接使用，也适合作为 AI Agent 的调用基础

可以进一步细分为两类：

- 单节点部署包：解决某一个业务动作
- 链路部署包：把多个单节点串成一个可跑通的最小闭环

## 当前公开分发建议

当前不做网站内下单，也不把“是否进私有仓库”作为主路径。

推荐 V1 流程：

1. 用户在 `brand-site` 或 GitHub 看到工具目录和组合包说明
2. 用户直接下载或 clone 公开仓库
3. 用户按文档运行单节点 Skill 或链路组合包
4. 用户基于 agent 环境自行替换样例或继续改造
5. 后续再通过课程、答疑或联系入口承接更深入需求

当前推荐的公开资源形式有两种：

### 形式 A：公开 GitHub 仓库

适合：

- 会使用 GitHub 的用户
- 需要持续收更新的用户
- 需要看源码和继续改造的用户
- 拥有 agent 环境、希望直接安装运行的学员

优点：

- 更新方便
- 安装路径清晰
- 可持续迭代

### 形式 B：公开下载包 / ZIP

适合：

- 不想直接碰 GitHub 的用户
- 只想拿部署包和使用说明的用户
- 偏课程型、执行型用户

建议内容：

- ZIP 包
- 安装说明
- 使用样例
- 更新说明

## 当前最适合的公开资源形式

结合你现在的目标，建议并行保留两种：

- 对开发者型用户：公开 GitHub 仓库
- 对普通学员：公开下载包 / ZIP

两者都可以由 `brand-site` 的说明页承接，但核心分发路径先以“公开可获取”为准。

## 文案与页面策略建议

公开工具页面的核心表达是：

- 这个工具已经开源
- 你可以单独拿去用
- 如果你想直接跑完整闭环，可以进入组合包目录

后续承接页面的核心表达是：

- 公开仓库先解决安装、运行和复用
- 后续扩展层再解决课程承接、案例深化和更深服务

## 当前建议的最小落地动作

1. 在 `brand-site` 增加“工具 / Skills”总入口
2. 为当前 4 个公开单节点分别做详情页
3. 新增一个“主动开发最小闭环链路组合包”介绍页
4. 明确标注当前组合包已公开可安装
5. 页面中统一加入微信 / 公众号联系入口

## 与母目录方法论的关系

这套分发方案符合当前母目录原则：

- 单节点 Skill 继续作为默认公开发布单元
- 链路组合包作为第二层，优先服务课程演示、公开安装、业务闭环验证
- 不直接跳到总 Skill / 总编排器
- 不重写现有节点逻辑，而是先做好衔接、说明和交付形式

## 当前结论

现阶段最稳的策略不是让 `brand-site` 直接做会员后台，而是：

- 用 `brand-site` 做工具说明前台和承接入口
- 用公开单节点建立开源入口和内容传播
- 用公开链路组合包降低安装门槛、扩大使用反馈
- 后续再决定会员扩展层怎么承接

后续如果用户规模扩大，再决定是否追加课程扩展、下载分层、会员后台或自动发放机制。
