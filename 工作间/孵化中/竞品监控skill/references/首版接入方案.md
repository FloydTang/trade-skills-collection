# 竞品监控 Skill 首版接入方案

## 顶层判断

`changedetection.io` 不适合作为我们要重写的对象，更适合作为首版上游监控底座。

当前更合理的拆分是：

- `changedetection.io`：负责抓取、比对、定时检查、告警能力
- `竞品监控 Skill`：负责业务输入结构、关注字段定义、变化摘要、人工复核口径

## 为什么这样拆

- 底层监控能力已经成熟，自己重写性价比低
- 你当前真正缺的是“怎么把变化讲成业务动作”
- 这样更符合“实事求是、简洁高效”

## 首版流程

1. 选 3 个真实竞品页面
2. 为每个页面定义一个明确的关注目标
3. 在 `changedetection.io` 里建立 watch
4. 只对关键字段使用选择器或过滤规则
5. 获取变化结果
6. 输出一份适合人工复核的变化摘要
7. 把稳定的判断沉淀到 `工作间/`
8. 如该说明未来会被课程、品牌、产品表达反复使用，则在 Obsidian `工具工作间/外贸skill` 保持可见
9. 决定是否进入销售讨论、市场复查或需求池增强

## 当前首版监控对象

- `Allbirds / 产品详情页`
  - 页面：`https://www.allbirds.com/products/mens-tree-runners`
  - 关注字段：产品标题、价格、规格关键词、CTA 文案
- `Allbirds / 新品集合页`
  - 页面：`https://www.allbirds.com/collections/mens-new-arrivals`
  - 关注字段：首屏主文案、主推产品、上新标识、CTA 文案
- `Allbirds / 首页促销区`
  - 页面：`https://www.allbirds.com/`
  - 关注字段：促销文案、折扣文案、CTA 文案

这样先用同一竞品的 3 类页面跑通，是因为字段口径更统一，方便先验证“变化抓取 -> 变化摘要 -> 人工复核建议”。

## 首版建议关注字段

- 产品标题
- 价格
- 规格关键词
- 折扣/促销文案
- CTA 文案
- 上新标识

## 输出建议

每次变化摘要至少包含：

- 哪个竞品
- 哪个页面
- 变化类型
- 变化内容简述
- 可能业务含义
- 是否需要人工复核
- 建议下一步动作

## 当前不建议马上做的事

- 不先做全站自动发现
- 不先做几十个竞品的批量监控
- 不先做复杂 BI 或数据库报表
- 不把通知链路做成过长流程
- 不为了凑完整度继续加低价值字段

## 后续可扩展方向

- 增加 JSON API 监控
- 增加 PDF 监控
- 增加价格变化专项模板
- 增加周报 / 月报汇总格式
